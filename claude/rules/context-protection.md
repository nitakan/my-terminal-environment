# コンテキスト保護とサブエージェント委任

## 目的（Why）

メインセッションのコンテキストを汚さないことが目的。サブエージェントは独立したコンテキストウィンドウを持ち、メインスレッドには**最終要約だけ**が返る。重い読み込み・探索・実装をサブエージェントへ逃がすことで、メインのコンテキストを意思決定に集中させ、長いセッションでも劣化・compaction を遅らせる。

## メインセッションの役割

メインセッションは**オーケストレーション**に徹する：タスク分解、サブエージェントの起動・管理、結果の統合、ユーザーとの対話、最終判断。

メインで直接やらないこと（コンテキストを浪費するため）：
- 大きなファイルの通読、広範なコード検索結果の生展開
- `git diff` / `git show` など差分の生出力
- 調査・設計・実装・まとまった作業の本体

## 委任の原則（What）

調査・設計・実装・作業はサブエージェントで実施する：
- 調査・探索 → `Explore` / `general-purpose`
- 設計 → `Plan`
- 実装・まとまった作業 → `general-purpose`

注意点：
- サブエージェントは独立コンテキストのため、プロンプトは**自己完結**させる（背景・対象パス・制約を明記）
- 返答は**結論・要約のみ**を求め、生ログやファイル全文を持ち帰らせない
- 独立タスクは1メッセージで**並列起動**する
- 重い実装（codexループ・複数test）は `run_in_background` で回す

## モデル選定（How）

- 設計 → **Opus** を原則
- その他（調査・レビュー・軽作業）→ **Sonnet** を原則
- 実装 → 難易度に応じて **Opus / Sonnet** を適宜選択

## 例外（委任しない）

オーバーヘッドが委任の利得を上回る場合はメインで直接対応してよい：
- 単一事実の確認、1ファイルの該当箇所だけの参照
- 小さな設定・メタ編集
- ユーザーとの対話・質問応答
